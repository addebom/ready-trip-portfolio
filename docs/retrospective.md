# Retrospective - 준비해여행

## 프로젝트를 시작하게 된 이유

준비해여행은 거창한 아이디어에서 시작한 프로젝트가 아니었습니다.

여행을 준비할 때마다 준비물을 다시 정리해야 했고, 출발 직전에 무언가를 빠뜨렸는지 불안해하는 경험이 반복되었습니다.

기존에도 메모장이나 메시지 앱을 사용해봤지만, 여행 준비에 특화된 구조는 아니었기 때문에 결국 매번 같은 과정을 반복하게 되었습니다.

그래서 여행 준비라는 반복적인 행동을 조금 더 체계적으로 관리할 수 있는 서비스를 만들어 보고 싶었습니다.

---

## 개발보다 어려웠던 것

프로젝트를 시작할 때는 기능을 구현하는 것이 가장 어려울 것이라고 생각했습니다.

하지만 실제로는 어떤 기능을 만들지 결정하는 과정이 더 어려웠습니다.

처음에는 다양한 기능을 넣고 싶었습니다.

* 여행 일정 관리
* 국가별 정보 제공
* 환율 정보
* 여행지 추천
* 쇼핑 기능

하지만 기능을 추가할수록 서비스의 목적이 흐려진다는 것을 느꼈습니다.

결국 MVP 단계에서는

"준비물을 빠뜨리지 않도록 돕는다"

라는 핵심 문제에 집중하기로 결정했습니다.

---

## 출시 후 가장 크게 달라진 생각

프로젝트를 진행하면서 가장 큰 변화는

"개발 완료 = 프로젝트 완료"

가 아니라는 점을 알게 된 것입니다.

출시 전에는

서비스를 만드는 것이 목표였습니다.

하지만 출시 후에는

사용자가 어떻게 행동하는지 관찰하는 것이 더 중요해졌습니다.

실제 사용자가 서비스를 이용하기 시작하면서

* 어떤 기능을 사용하는지
* 어떤 화면에서 머무르는지
* 어떤 기능은 사용하지 않는지

에 관심을 가지게 되었습니다.

서비스를 만든 이후부터가 진짜 시작이라는 것을 처음 경험했습니다.

---

## 잘했다고 생각하는 점

### 문제를 단순하게 유지한 점

많은 기능을 넣기보다 여행 준비라는 핵심 문제에 집중했습니다.

덕분에 서비스 구조가 복잡해지지 않았고, 사용자가 기능을 이해하기 쉬운 형태를 유지할 수 있었습니다.

### 실제로 출시한 점

학습용 프로젝트로 끝내지 않고 실제 서비스 형태로 출시했습니다.

작은 규모였지만 실제 사용자가 서비스를 사용하는 경험을 관찰할 수 있었고, 이는 개발 과정에서 얻기 어려운 경험이었습니다.

### 사용자 관점에서 생각하려고 노력한 점

개발자 입장에서 기능을 추가하기보다 사용자가 어떤 상황에서 사용할지를 먼저 고민했습니다.

특히 출국 준비와 귀국 준비를 분리한 것은 사용자 입장에서 생각한 결과였습니다.

---

## 아쉬웠던 점

### 사용자 검증 부족

서비스를 만들면서 스스로는 편리하다고 생각했지만, 충분한 사용자 인터뷰나 검증 과정을 거치지는 못했습니다.

앞으로는 기능을 만들기 전에 실제 사용자의 의견을 먼저 확인하는 과정이 필요하다고 생각합니다.

### 운영 데이터 부족

사용자가 존재하기는 했지만 아직은 데이터를 분석할 수 있을 만큼 충분한 규모는 아니었습니다.

향후에는 사용자의 행동을 더 구체적으로 측정할 수 있는 구조를 고민해보고 싶습니다.

### 기능 우선순위 판단 경험 부족

개발 과정에서 여러 아이디어가 계속 떠올랐지만 어떤 기능을 먼저 구현해야 하는지 판단하는 경험이 부족했습니다.

서비스를 운영하면서 기능을 추가하는 것보다 어떤 기능을 만들지 결정하는 것이 더 중요하다는 점을 느꼈습니다.

---

## 가장 기억에 남는 경험

실제 사용자가 서비스를 이용했다는 사실입니다.

사용자 수 자체가 중요한 것은 아니었습니다.

처음으로

"내가 만든 서비스를 누군가가 실제로 사용한다"

는 경험을 할 수 있었던 것이 가장 의미 있었습니다.

이 경험을 통해 프로젝트를 보는 관점도 바뀌게 되었습니다.

---

## 다음에 다시 만든다면

다음 버전에서는 새로운 기능을 추가하기보다 사용자의 행동을 더 잘 이해하는 방향으로 개선하고 싶습니다.

예를 들어

* 여행 유형별 체크리스트
* 반복 사용자용 템플릿
* 국가별 준비물 추천
* 사용자 피드백 수집 기능

등을 검토해보고 싶습니다.

---

## 프로젝트를 통해 배운 점

준비해여행은 기술적으로 가장 복잡한 프로젝트는 아니었습니다.

하지만 문제를 정의하고, 서비스를 설계하고, 실제로 출시하고, 사용자를 관찰하는 전 과정을 경험할 수 있었던 프로젝트였습니다.

이 프로젝트를 통해

기능을 만드는 개발자보다

문제를 해결하는 서비스 개발자가 되고 싶다는 생각을 하게 되었습니다.
